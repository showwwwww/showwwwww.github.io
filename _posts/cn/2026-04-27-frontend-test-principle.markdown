我理解前端测试时，需要遵守的原则其实很简单：`UI = f(state)`。组件的界面由它接收和维护的状态计算出来。接受这个前提之后，前端单元测试会变得清晰很多：我不需要证明屏幕上的每一个像素都正确，而是要证明状态能按预期变化，并且 React 会在正确的时机基于这些状态重新渲染。

![以状态为中心的前端测试模型](/assets/img/posts/frontend-test-principle/state-first-frontend-tests.png)

## 心智模型

现代前端系统经常被 UI 描述，因为用户最终看到的是界面。但屏幕上的 UI 只是整个链路的最后一步。

更稳定、更值得测试的东西，是内存里的数据结构：props、本地 state、store state、缓存数据、派生 selector，以及事件回调里的 payload。一次点击、一次键盘输入、一次网络响应，或者一次拖拽手势，重要的地方不在于它真的发生在页面上，而在于它改变了这份数据结构。

所以我更愿意这样理解前端行为：

```text
当前状态 + 用户动作 + 外部结果 -> 下一个状态
下一个状态 -> 渲染输出
```

第一条箭头是业务行为，第二条箭头是 React 的渲染工作。在大多数单元测试里，第一条箭头更值得关注。

## 好的单元测试应该证明什么

对组件级单元测试来说，我通常想确认三件事。

第一，初始状态是正确的。组件应该从一个明确的数据形状开始。如果初始状态就是错的，后面的所有交互都会变得难以推理。

第二，每一次交互都能产生正确的下一个状态。当用户输入、选择、提交、展开、关闭、重试或取消时，测试应该验证真正重要的状态值。这个状态可以是 React 本地 state、reducer 的返回值、store 更新、query cache 的变化，也可以是回调函数收到的 payload。

第三，当状态契约要求重新渲染时，React 确实被要求重新渲染。有时这意味着检查某个 render function 被调用了；有时意味着检查子组件收到了新的 props；有时也意味着检查一个昂贵组件没有因为无关状态变化而重新渲染。渲染次数不总是产品需求，但当组件很重，或者无意义渲染曾经造成过真实问题时，它可以成为一个有价值的性能契约。

如果这三件事都成立，我通常就可以认为这个组件在单元测试层面的行为是健康的。

## 我不会过度测试什么

我不想在单元测试里花太多精力证明浏览器的渲染引擎。如果 React 收到了正确的状态，组件也返回了正确的 element tree，我不需要再通过单元测试证明浏览器会绘制一个 `div`、应用普通 CSS，或者派发一次标准 click 事件。

这并不是说 DOM 测试没有价值，而是我希望明确自己为什么要使用它。可访问性语义、焦点行为、表单提交、portal 位置、依赖布局的逻辑，以及浏览器集成问题，都可能值得写 DOM 层面的测试。但这些测试应该是因为 DOM 本身就是风险点，而不是因为每一个组件测试默认都要检查 DOM。

问题在于，过度依赖 DOM 的单元测试很容易绑定到偶然的结构上。多一层 wrapper、改一个 class name，或者挪动一段无关紧要的 markup，都可能让测试失败。这种失败不一定说明产品坏了，也可能只是说明测试盯错了对象。

## 一个实际的测试形状

写前端单元测试时，我喜欢先问一个问题：

> 我到底想证明哪一次状态转移？

这个问题通常会把测试带到更清楚的结构里。

```text
given：一个已知状态
when：发生一个用户动作或外部事件
then：关键状态变成预期的值
and：渲染契约被满足
```

比如一个筛选面板的测试，不一定要从「checkbox 在屏幕上看起来是不是 checked」开始。它可以从「用户选择一个筛选项之后，selected filter state 里是否包含正确的值」开始。之后我可以继续验证列表组件是否拿到了预期的筛选后数据，或者负责渲染列表的组件是否用正确的 props 被调用。

这样测试会更接近真正的行为。checkbox 只是改变筛选状态的一种界面形式，状态转移才是产品规则。

## 边界在哪里

集成测试和 E2E 测试仍然有自己的位置。一个前端产品不只是 reducer 和 render function。真实用户关心路由、网络时序、CSS、可访问性、浏览器 API，以及可见的反馈。

但我不希望每一个单元测试都承担所有这些重量。单元测试应该让状态机变得可信；集成测试应该证明几个模块能正确协作；E2E 测试应该证明最重要的用户路径在真实浏览器里可以跑通。

层次不同，是因为风险不同。

## 原则

我的原则不是「永远不要测试 DOM」。我的原则是：

> 在单元测试里，先把前端行为当成状态转移来测试。把 DOM 渲染视为输出契约，只有当 DOM 本身就是风险点时，才深入检查它。

这个心智模型会让测试更小、更稳定，也更贴近现代前端系统的架构。用户交互是输入，状态变化是核心行为，React 渲染是把状态投射成 UI 的过程。

当状态正确变化，并且渲染契约在预期时机被触发时，我就已经测试了前端代码真正由我负责的那一部分。
